## Summary

Daily run of the smoke-test + QA suite surfaced three stale assertions that
no longer match the current API contract. This PR fixes those, expands smoke
coverage to endpoints added since the last sweep, and adds a new
`daily-review-2026-04-30.test.ts` Jest file pinning today's contract surface.

## Bugs fixed

1. **`scripts/qa/smoke-test.ts` — search 200/400 expectations are stale.**
   `/api/properties/search` now sits behind `requireAuth` +
   `enforceFreeUsageLimit('property_search')` and uses the
   `address|zip|city|placeId|parcelId` schema (not `?query=`). The smoke run
   was failing with `expected 200, got 401` and `expected 400, got 401`.
   Updated to expect 401 for both anonymous probes and reworked the
   property-detail / risk / insurance / insurability / carriers chain to
   only run when an explicit `--property-id <uuid>` flag is supplied.
2. **`scripts/qa/health-check.ts` — same stale expectation.**
   Health check asserted 400 from `/api/properties/search` with no params;
   now expects 401.
3. **No public smoke coverage for liveness/static endpoints.**
   `/health` and `/robots.txt` (the latter was added in the 2026-04-12
   review) had no smoke or health checks. Added both.

## Coverage gaps closed

- `GET /` (API metadata)
- `GET /health` (liveness)
- `GET /robots.txt` (disallow-all body assertion)
- `GET /api/properties/suggest?q=Mia` (200 success path)
- `GET /api/properties/suggest?q=a` (400 short-query path)
- `GET /api/dashboard/ticker` → 401 unauthenticated
- `GET /api/deals` → 401 unauthenticated
- `GET /api/alerts/carrier-exits` → 401 unauthenticated
- `GET /api/push/vapid` → 200 (configured) or 503 (graceful degrade)

## New unit tests (`daily-review-2026-04-30.test.ts`)

- `SearchQuerySchema` — coerce `page`/`limit`, reject malformed `state`/`zip`,
  reject `limit > 50`.
- Properties `/search` auth contract — encodes the documented guard order
  (`requireAuth` → `enforceFreeUsageLimit`) and the 401-not-400 anonymous
  status so a regression flips a failing test.
- Alerts router schemas — `severity` enum, paginated `limit` parsing, ack
  param length bounds.
- Deals router enums — 5 pipeline stages, 11 fallout reasons (incl.
  `INSURABILITY`).
- Notifications push subscribe schema — endpoint URL, key non-emptiness,
  user-agent length cap.
- CORS `isOriginAllowed` — allow exact match, allow `(www|api|app).coverguard.io`,
  allow Vercel preview hosts, block lookalikes (`coverguard.io.evil.com`),
  block excessively long origins, block other subdomains.
- `publicAppUrl` precedence — `APP_PUBLIC_URL` > `NEXT_PUBLIC_APP_URL` >
  hard-coded `https://coverguard.io`.

## Why this didn't open a PR automatically

This run was a scheduled, unattended task. The runtime sandbox does not have
read access to the repo's `.git/config`, so `git checkout -b` /
`git push` cannot complete from here. The branch + PR open step needs to be
done from a workstation:

```bash
cd "/path/to/CoverGuard Repo Clone/Main"
git checkout -b claude/daily-qa-2026-04-30
git add scripts/qa/smoke-test.ts scripts/qa/health-check.ts \
        apps/api/src/__tests__/daily-review-2026-04-30.test.ts \
        docs/daily-qa/2026-04-30-pr-draft.md
git commit -S -m "chore(qa): refresh smoke + health-check, add 2026-04-30 daily-review tests"
git push -u origin claude/daily-qa-2026-04-30
gh pr create --fill   # or open the compare URL in Chrome
```
